# Repository Guidelines

## Project Shape

Freely Spoken is an Expo/React Native iOS app with a FastAPI lookup backend. The repository name is still `mic-check`, but the implemented product is Freely Spoken, with `christian` and `dhammapada` variants and a stubbed `stoic` variant. The Dhammapada variant ships as Idle Ashes and is currently in TestFlight.

Read `README.md` first for the privacy model, architecture, setup, and repository map. Read `docs/privacy-policy.md` before changing any data collection, logging, retention, network behavior, or wording about privacy.

## Privacy Boundary

The core product promise is that audio and raw transcripts stay on device. The backend may receive only:

- `appVariant`
- `anonymizedText`
- `sentiment`
- `emotions`
- `confidence`

Do not add audio, raw transcript text, audio paths, recording duration, device identifiers, account/session identifiers, or location to outbound app requests, backend logs, tests, fixtures, or debugging output. The request allowlist lives in `services/lookup-request.ts`; update its tests whenever that contract changes.

Treat local Foundation Models output as untrusted. Keep parsing, normalization, and anonymization guards defensive, and avoid fallback paths that can leak source wording.

## App Work

- Use TypeScript strict mode and existing Expo Router patterns.
- Keep reusable UI in `components/`, app flow in `app/`, product tokens in `constants/`, and side-effectful pipeline logic in `hooks/` or `services/`.
- The main mobile flow is in `app/index.tsx`: `idle -> recording -> processing -> review -> responseLookup -> results`.
- Hooks for the pipeline should keep the existing shape: `{ result, isLoading, error, action, reset }`.
- Do not make durable product changes inside generated native folders. `ios/` and `android/` are generated by Expo prebuild and are ignored by git; put durable native identity and permissions changes in `app.config.js`, `app.json`, assets, or config plugins.
- This app cannot run in Expo Go. Full recording, Apple Speech, and Apple Foundation Models behavior requires a native iOS build on capable Apple hardware.

## Backend Work

- Backend code lives under `server/app`; tests live under `server/tests`.
- `server/app/main.py` should stay a thin HTTP layer. Provider fallback belongs in `server/app/llm_runner.py`; variant behavior belongs in `server/app/lookup/`.
- The backend should log operational metadata only. Do not log `anonymizedText` or any user-provided body text.
- LLMs select references or catalog IDs. They must not author canonical Bible or Dhammapada text.
- When changing prompts or lookup adapters, check whether the local lookup harness in `tools/lookup-harness/` needs the same change.
- Dhammapada runtime behavior must use `server/app/lookup/dhammapada_catalog.json`; tooling under `tools/dhammapada-labeling/` is development-only.

## Variants

The active product is selected at build time with `EXPO_PUBLIC_APP_VARIANT`.

- `christian`: Freely Spoken, implemented.
- `dhammapada`: Idle Ashes, implemented and in TestFlight.
- `stoic`: unreleased stub.

`app.config.js` owns variant-specific native identity, including display name, bundle identifiers, URL schemes, icons, and permission string product names. Do not put per-variant identity back into `app.json`.

## Commands

Use these checks for app changes:

```bash
npm run lint
npm run typecheck
npm test
```

Use these checks for backend changes:

```bash
cd server
pip install -e '.[dev]'
pytest
```

Useful local run commands:

```bash
npx expo run:ios --device
EXPO_PUBLIC_APP_VARIANT=dhammapada npx expo run:ios --device
cd server && uvicorn app.main:app --reload --port 8080
cd tools/lookup-harness && ./start.sh
```

## Secrets And Generated Files

Never commit `.env` files, provider keys, Apple credentials, provisioning files, local sample output, or generated build artifacts. Use `.env.example`, `server/.env.example`, and `tools/lookup-harness/.env.example` as documentation only.

`EXPO_PUBLIC_LOOKUP_CLIENT_SECRET` is a build-time client gate, not a true server secret; assume mobile binary values are recoverable.

## Tests To Update

- Privacy request shape: `services/__tests__/lookup-request.test.ts`
- Sentiment/anonymization parsing and guards: `hooks/__tests__/use-sentiment-analyzer.test.ts`
- Backend HTTP, crisis, provider, and lookup behavior: `server/tests`
- Dhammapada shortlist/catalog/validation behavior: `server/tests/test_dhammapada_*.py`

Prefer focused tests around changed contracts and privacy-sensitive behavior. Backend tests should remain hermetic; stub provider calls rather than requiring API keys or network access.
